  • links
  • three parts
    • (Initialization)

      set up state and props

      usually done with constructor

    • Mounting - birth

      component mounts to the DOM (created and inserted)

      • methods available

        componentWillMount()

        componentDidMount()

    • Update - growth
      • methods available

        shouldComponentUpdate()

        componentWillUpdate()

        componentDidUpdate()

    • Unmount - death

      componentWillUnmount()

  • Common methods
    • what is a pure method?

      method that does not have any side-effects and will always return the same output when the same inputs are passed

      ^ can't/shouldn't use Math.random()

    • render()

      creates the actual component

      only one required

      you cannot setState() in render!!

    • componentDidMount()
      • can it use setState()?

        yes, but use with caution bc performance issues

        it will trigger another render() but will happen before it's shown on the browser

      • when is it called?

        as soon as the component is mounted and ready

    • componentDidUpdate()
      • when is it called?

        as soon as updating happens

      • can it use setState()?

        yes, should be in a conditional to check for state/prop changes

        could lead to an infinite loop if used improperly

      componentDidUpdate(prevProps) {
 //Typical usage, don't forget to compare the props
 if (this.props.userName !== prevProps.userName) {
   this.fetchData(this.props.userName);
 }
}
    • componentWillUnmount()
      • when is is called?

        right before the component is unmounted/destroyed

      • can you call setState()?

        no: it will never be re-rendered

      • things you can/should do

        clean up.

        clear timers, cancel api calls, clear caches, etc

  • Uncommon methods
    • shouldComponentUpdate()
      • can you update setState?

        no

      • when is it called?

        when a component rerenders

      • purpose?

        tell react is a component is not affected by the state and prop changes

        really only meant for some performance optimization

      • caution for use

        can lead to several bugs

      //note that it should return a boolean variable
shouldComponentUpdate(nextProps, nextState) {
 return this.props.title !== nextProps.title || 
  this.state.input !== nextState.input }
    • static getDerivedStateFromProps()
      • alternative to..?

        previous lifecycle method componentWillRecieveProps()

      • when is it called?

        just before render()

      • does it have access to this

        no

      • what does it do?

        returns an object to update state in response to prop changes

      static getDerivedStateFromProps(props, state) {
    if (props.currentRow !== state.lastRow) {
      return {
        isScrollingDown: props.currentRow > state.lastRow,
        lastRow: props.currentRow,
      };
    }
    // Return null to indicate no change to state.
    return null;
  }
    • getSnapshotBeforeUpdate()
      • when is it called?

        right before the DOM is updated

      value from this method is passed to componentDidUpdate()

      • should is be used?

        rarely, if at all
